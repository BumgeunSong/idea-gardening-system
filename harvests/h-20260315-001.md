Q: 🌱 자연어 코드베이스 쿼리를 위한 용어 사전 기획 — 이 주제에 대해 어떤 생각이 있어?

A: # 자연어 코드베이스 쿼리를 위한 용어 사전 기획
# 목적
협업 과정에서 비개발자가 개발자에게 현재 앱의 동작을 질문하는 경우가 많다. 비개발자가 도메인 용어로 코드베이스를 질문할 수 있게 한다. 그렇게 해서 팀의 커뮤니케이션 비용을 줄이고 싶다.
예시 질문:
* "1원인증 이후에 출금이체동의를 받는 퍼널로 연결돼?"
* "외며들기(ForeignerPlore)에서 토스뱅크 EDD가 끝나면 유저는 어떤 화면을 보게 돼?"

⠀LLM이 이 질문을 이해하려면 두 가지가 필요하다. 도메인 용어가 무엇인지(1원인증 = 계좌 소유 확인 절차), 그리고 그게 코드에서 어떤 식별자로 존재하는지(one_won_verification, WonVerifyFunnel). 지금은 이 매핑이 팀원 머릿속에만 있어서 쿼리 정확도가 낮다.
# 왜 일반 시맨틱 검색으로는 안 되나
기존 코드 시맨틱 검색 도구들(Sourcegraph, grepai 등)은 쿼리와 코드가 같은 어휘를 쓴다는 전제가 깔려 있다. "authentication"을 검색하면 authenticate() 함수를 찾는 식이다.
Toss의 문제는 다르다. "1원인증"이라는 단어는 코드에 없다. 코드엔 one_won_verification으로 존재한다. 쿼리와 코드베이스가 **다른 언어**를 쓰는 것이다. 벡터 유사도로는 이 간극을 메울 수 없다 — 사내 용어는 수십억 문장에서 학습된 Embedding 모델이 알 리 없으니까.
# 핵심 개념: Semantic Glossary
도메인 용어들을 단순히 나열하는 게 아니라, **용어 간 관계(선행조건, 후속단계 등)와 코드 식별자 매핑, 그리고 다양한 표기 변형(alias)까지 함께 저장**하는 구조.
내부적으로는 **Knowledge Graph**다. 노드는 도메인 개념이고, 링크는 이름 붙은 관계(선행조건, 후속단계, 담당팀)다. 링크에 이름이 있기 때문에 "이후에"라는 쿼리 의도에 맞는 링크만 골라서 탐색할 수 있다.
LLM이 코드를 검색하기 전에 이 Glossary를 먼저 조회해서, 자연어를 실제 코드 식별자로 번역한 뒤 검색한다.
### 자연어 질문
###   → Entity Linking (용어 추출 + Knowledge Graph 노드 연결)
###   → 코드 식별자로 번역
###   → 코드베이스 검색
###   → 답변 생성
# 결정 사항
**저장 형식: Markdown + YAML frontmatter**
* 파일 하나 = 용어 하나
* LLM-friendly: 파싱 없이 컨텍스트에 바로 투입 가능
* 사람이 편집하기 좋음: GitHub PR, Obsidian 등으로 관리 가능
* related 필드가 Knowledge Graph의 엣지 역할 → 나중에 그래프 DB로 마이그레이션할 때 구조 재사용 가능

⠀**Alias 구축 전략**
팀원마다 같은 개념을 다르게 부른다("1원인증", "1원 인증", "원화인증"). 이 변형들을 alias로 등록해야 Entity Linking이 작동한다. 세 단계로 점진적으로 구축한다:
**1** **코드베이스 스캔**: Claude Code로 식별자를 추출하고 LLM으로 그룹핑 → 초기 Glossary 자동 생성
**2** **비구조화 텍스트 수확**: Slack, PR 코멘트, 위키에서 alias 후보 추출 → 사람이 검토
**3** **사용 로그 강화**: 쿼리할 때마다 연결 결과를 기록 → 교정이 새 alias로 누적

⠀**시작 방법**
팀원 인터뷰 대신, Claude Code로 코드베이스 초안을 먼저 뽑고 팀원이 검토·수정하는 방식으로 시작한다. 빈 종이보다 틀린 초안 고치는 게 빠르다.

Q: 이 프로젝트에서 가장 기대되는 부분이 뭐야?

A: 업무를 할 때 팀원들 (비개발자&lt;&gt;개발자, FE개발자&lt;&gt;서버개발자)이  질문하고, 답변을 기다리는 의존 관계 (or 커뮤니케이션 비용)를 줄이고 싶어. 그렇게 하면 전체 팀의 업무 속도가 올라갈거라고 믿어.

Q: 그 의존 관계를 줄이는 게 왜 그렇게 중요하다고 생각해?

A: • 지식 노동자의 비생산 시간 중 상당 부분은 커뮤니케이션과 정보 대기에서 발생하고, 이는 개인의 처리 속도가 아니라 의존 구조의 문제다. 
    ◦ *처리 시간(processing time)* — 실제로 일하는 시간
    ◦ *대기 시간(waiting time)* — 다른 사람의 아웃풋을 기다리는 시간
• Eliyahu Goldratt의 책 &lt;더 골(The Goal&gt;을 보면 이런 아이디어가 나온다. 
• 혼자 하는 일의 변동은 국소적으로 흡수된다. 근데 의존 관계가 생기는 순간, 앞 단계의 변동이 *증폭되어* 뒤로 전파된다.
• TOC(Theory of Constraints, 제약 이론)의 병목 이론을 적용하면, 이 의존 관계를 줄이는 것이 개인 속도 향상보다 시스템 처리량에 더 큰 레버가 된다.
• 의존 관계 제거/비동기화는 대기 시간을 줄이고, 지수적, 시스템적인 효과를 가져와.
• "내가 더 빠르게 일한다"가 아니라 "동료가 나를 기다리지 않아도 되는 구조를 만든다." 이것이 임팩트가 더 크다.

Q: 이런 생각이 생긴 계기가 뭐야? 특별한 경험이 있었어?

A: • 실제로 개발하다보면, 내가 개발을 느리게 해서 제품이 늦게 나오는 것보다, 무언가를 다른 작업이 완료되기를 기다리거나 다른 사람의 결정을 확인해야하거나, 피드백이 일찍 오지 않아 후속 단계에서 다시 돌아가서 고치는 시간. 이런 것들이 제품 출시를 늦추는 경우가 더 많았기 때문이야. 

Q: 이 경험과 너의 다른 관심사들(Product Engineer의 정체성, AI 시대의 인간 역량)이 어떻게 연결되는 것 같아?

A: • AI가 고급 추론이나 암묵적 컨텍스트가 많은 일은 잘 못할 수 있어.
• 그리고 대부분 AI가 소프트웨어 엔지니어링을 잘한다고 할 때는 처리 시간에 초점을 두고 있어.
• 하지만 AI의 능력이 실제로 팀의 생산성을 높일 수 있는 레버리지 (= 대기 시간)에 투입된다면 굉장히 효과는 커. 
• 예를 들어, 간단한 코드베이스 확인 질문은 크게 어려운 게 아닌 반면 AI는 24시간 대답이 가능하기 때문에 대기 시간을 확 줄일 수 있지.
• 물론, 코드베이스 확인 질문 또한 전혀 사람의 개입 없이 신뢰성 있는 답을 얻으려면 보완 시스템이 반드시 필요해. 그런 시스템 중 하나가 내가 위에서 말한 '자연어 코드베이스 쿼리를 위한 용어 사전'이야.
