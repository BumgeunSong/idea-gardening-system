Q: 🌱 StrongDM이 software factory에서 사용한 구현 검증 시스템 — 이 주제에 대해 어떤 생각이 있어?

A: # StrongDM 검증 시스템: 기존 테스트와 무엇이 다른가

&gt; “테스트 코드가 아니라 사용자 스토리로 검증한다.”
-----

## 배경: 왜 새로운 검증 방식이 필요한가

AI 코딩 에이전트가 코드를 작성하는 시대가 됐다.  
그런데 기존 테스트 방식에는 치명적인 구멍이 있다.

&gt; AI가 구현도 하고, 테스트 코드도 같이 수정할 수 있다.
```
// AI가 이렇게 해버릴 수 있다
function transfer(amount) {
  return true; // 테스트 통과용
}
```

`return true` 한 줄로 테스트는 초록불이 된다.  
StrongDM은 이 문제를 근본적으로 풀기 위해 검증 시스템 자체를 재설계했다.

-----

## 기존 Testing Pyramid와의 비교

### 기존 Testing Pyramid

```
        [E2E Tests]          ← 느리고, 유지비 높고, flaky
       [Integration]         ← 범위 제한적
      [Unit Tests]           ← 빠르지만 AI가 함께 수정 가능
```

**구조적 한계**

• 모든 테스트가 코드베이스 **안**에 존재한다
• AI가 구현과 테스트를 **동시에** 수정할 수 있다
• E2E는 경로가 고정돼 있어 UI가 조금만 바뀌어도 깨진다 (flaky)
• 커버리지를 높이려면 테스트 코드도 그만큼 늘어난다
• “테스트가 초록불” = “소프트웨어가 동작한다”는 보장이 없다
-----

### StrongDM의 검증 시스템

```
[Scenario 파일] ── 코드베이스 밖, AI 접근 불가 (holdout set)
      │
      ▼
[Digital Twin] ── 외부 서비스의 API 동작을 복제한 가짜 서버
      │
      ▼
[User Agent] ── Playwright 등으로 실제 유저처럼 행동
      │
      ▼
[Trajectory] ── "무엇을 했고, 무엇을 봤는가" 기록
      │
      ▼
[Judge Agent] ── LLM이 "이 유저가 만족했을까?" 확률로 판단
      │
      ▼
[Satisfaction Score] ── 전체 trajectory 중 몇 %가 유저를 만족시켰나
```

-----

## 세 가지 핵심 개념

### 1. Scenario — “코드베이스 밖의 holdout set”

ML 모델 훈련에서 test data를 training data와 분리하듯,  
Scenario는 구현 AI가 **절대 볼 수 없는 곳**에 저장된다.

```markdown
# scenario-001.md (별도 repo 또는 접근 제한 폴더)

사용자가 오늘 글을 작성한다.
→ 작성 후 streak이 1 증가해야 한다.
→ KST 자정 기준이 정확해야 한다.
→ 이미 작성한 날 다시 작성해도 streak이 중복 증가하지 않아야 한다.
```

자연어로 작성된 **목표**만 있다. 경로(어떤 버튼을 언제 클릭하는지)는 없다.  
AI가 Scenario를 볼 수 없으니 치팅이 불가능하다.

-----

### 2. Digital Twin — “관찰 가능한 동작만 복제한 가짜 서버”

기존 Mock과의 차이:

|             |Mock        |Digital Twin        |
|-------------|------------|--------------------|
|**누가 만드나**   |개발자가 가정해서 작성|AI가 API 문서에서 역공학    |
|**기준**       |개발자의 생각     |실제 서비스의 관찰된 동작      |
|**edge case**|개발자가 생각한 것만 |실제 에러 코드, 타임아웃 패턴 포함|
|**유지보수**     |사람이 계속 업데이트 |스펙 변경 시 AI가 재생성     |


&gt; **핵심**: 외부 서비스의 내부 로직을 복제하는 게 아니다.  
&gt; SDK가 그대로 꽂혔을 때 동작하는 수준의 **외부 인터페이스**만 복제한다.
실제 서버 없이도:

• Rate limit 걱정 없이 수천 번 실행 가능
• 위험한 failure mode (잔액 초과, 타임아웃) 안전하게 테스트
• API 비용 없음
-----

### 3. Satisfaction — “boolean이 아닌 확률적 판단”

기존 E2E: `expect(balance).toBe(20000)` → 통과/실패

Judge Agent: “이 trajectory를 봤을 때, 이 유저가 목표를 달성하고 불쾌하지 않았을 것 같은가?”

```
100번 실행 중 87번이 "만족" → Satisfaction 87%
```

**왜 확률인가?**

AI 에이전트는 매번 다른 경로로 목표에 도달한다.  
정해진 경로가 없는 소프트웨어는 boolean으로 검증할 수 없다.  
“무엇을 했는가”가 아니라 “유저가 원하는 걸 얻었는가”가 기준이다.

-----

## 기존 E2E와의 핵심 차이 요약

|            |기존 E2E           |StrongDM Scenario  |
|------------|-----------------|-------------------|
|**저장 위치**   |코드베이스 안          |코드베이스 밖 (holdout)  |
|**작성자**     |테스트 엔지니어         |사람 (자연어), 실행은 AI   |
|**판단 방식**   |Boolean (통과/실패)  |확률 (Satisfaction %)|
|**경로 정의**   |고정 (버튼 클릭 순서까지)  |목표만 정의             |
|**flaky 여부**|UI 변경 시 자주 깨짐    |목표가 바뀌지 않으면 안 깨짐   |
|**커버리지**    |사람이 케이스 추가해야 함   |AI가 다양한 경로 자동 탐색   |
|**AI 치팅**   |가능 (테스트 코드 같이 수정)|불가 (Scenario 접근 불가)|

## 장점

**1. AI 치팅 구조적 차단**  
Scenario가 코드베이스 밖에 있으면 AI가 구현과 검증을 동시에 조작할 수 없다.

**2. Flaky 문제 해결**  
경로가 아닌 목표로 검증하기 때문에 UI가 바뀌어도 Scenario는 깨지지 않는다.

**3. 유지보수 비용 급감**  
자연어로 작성된 Scenario는 코드 변경에 영향받지 않는다.  
기존 E2E처럼 “버튼 ID가 바뀌었으니 테스트 200개 수정”이 없다.

**4. 커버리지 확장 용이**  
Scenario 파일 하나 추가하면 된다. 코드가 아니라 문서 수준의 작업이다.

**5. 위험한 케이스 안전하게 테스트**  
Digital Twin으로 실제 서버에 위험한 요청을 보내지 않아도 된다.

-----

## 단점 및 현실적 한계

**1. Digital Twin 구축 비용**  
외부 서비스를 복제하는 초기 작업이 필요하다.  
다만 OpenAPI spec이 있으면 AI가 대부분 자동 생성할 수 있다.

**2. Satisfaction 판단의 신뢰성**  
LLM-as-judge도 틀릴 수 있다. Judge 프롬프트 설계가 중요하다.  
초기에는 사람이 Judge 결과를 모니터링하며 calibration이 필요하다.

**3. 완전한 시스템 구축까지 시간 필요**  
Scenario + Twin + User Agent + Judge를 모두 연결하는 파이프라인이 필요하다.  
기존 테스트 인프라를 당장 대체하기보다는 **병행 도입**이 현실적이다.

**4. 기존 결정론적 검증을 완전히 대체하지 않는다**  
잔액 계산처럼 정확히 맞아야 하는 로직은 여전히 기존 unit test가 효율적이다.  
Scenario 검증은 그 위에 추가되는 레이어다.

-----

## 우리 팀 적용 시 현실적인 시작점

전체 시스템을 한 번에 구축하지 않아도 된다.  
아래 순서로 점진적으로 도입할 수 있다.

```
1단계: Scenario 파일 작성
   → 코드와 분리된 폴더에 핵심 유저 스토리 3개 작성
   → 비용: 거의 없음

2단계: Digital Twin 구축
   → 우리가 호출하는 API의 OpenAPI spec으로 AI가 생성
   → Supabase처럼 공식 local 환경이 있으면 그걸 활용

3단계: User Agent 연결
   → Playwright 등으로 Scenario 자동 실행
   → Trajectory 기록

4단계: Judge Agent 추가 → Satisfaction 스코어 측정
```

## 한 줄 요약

&gt; 기존 테스트는 “코드가 내가 짠 대로 동작하는가”를 검증한다.  
&gt; StrongDM의 검증은 “유저가 원하는 걸 얻었는가”를 검증한다.  
&gt; 그 기준을 AI 손이 닿지 않는 곳에 두는 것이 핵심이다.

Q: 이 검증 시스템을 처음 접했을 때 가장 의외였던 점은 뭐야?

A: Digital twin이라는 개념. 나는 testing을 위해서는 고정된 mock 혹은 실제 서비스를 사용해야한다고 생각했어. 아직 확실히 그려진 건 아니지만, Digital twin은 그 중간인것 같아.

Q: Digital twin이 기존 mock과 실제 서비스의 중간이라고 느꼈는데, 이 접근법이 테스트의 어떤 근본적인 문제를 해결하려는 거라고 생각해?

A: Mock 기반의 테스트 - 모든 케이스를 다 사전 정의해야함. 실제 동작에 대한 결함 발견 커버리지가 낮음

실서비스 기반 테스트 - 테스트를 빠르고 예측가능하게 실행하기 어려움.

Q: 이 digital twin 방식이 실제로 우리 팀의 어떤 테스트 문제를 가장 잘 해결할 수 있을 것 같아?
