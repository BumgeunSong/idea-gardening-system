Q: 🌱 채점 기준의 중요성 — 이 주제에 대해 어떤 생각이 있어?

A: • 안드레 카파시가 만든 AutoResearch 프로젝트가 많은 주목을 받았다.
• *AI 에이전트가 스스로 모델 연구와 학습을 진행*하도록 설계
• AI 에이전트는 코드의 구조나 하이퍼파라미터, 옵티마이저 등을 자유롭게 수정
• 그런 다음 `val_bpb` (Validation bits per byte) 지표를 측정한다. 이 수치가 낮을수록 성능이 우수한 것이며, 어휘 사전의 크기(Vocab size)에 독립적인 지표이므로 아키텍처가 크게 변경되어도 공정한 비교가 가능합니다. AI는 이 지표를 바탕으로 코드 변경 사항을 채택할지 결정

Q: 그런데 왜 val_bpb 같은 독립적인 지표가 AI 연구 자동화에서 그렇게 중요해?

A: AI가 오래 제대로 돌기 위해서는 2가지가 중요하다.

첫째, '무엇이 되어야 하는지'를 얼마나 구체적으로 정의했는가.
둘째, '됐다/안됐다'를 AI가 스스로 검증할 수 있는가.

특히 둘째. 검증 시스템 차이가 크다.

예를 들어, git commit 메시지 컨벤션이 잘못됐다고 하자. AI는 뭐가 잘못인지 모른다. 기준이 없으니까. 이때 lint rule을 설정한다. 커밋할 때마다 검증하고, 기준 이하를 튕겨낸다면? AI는 화들짝 놀라서 수정한다. 내가 개입할 필요도 없이 피드백이 되먹여진다.

AI에게 채점 도구와 채점 기준을 줘야 한다.

쉽지는 않다. 모델 지능이 높아진다고 되는 문제도 아니다.

Lint, Test, 리뷰에 스타일과 스펙이 반영되어야 한다.
트레이드오프를 결정하기 위한 맥락이 있어야 한다.
디자인부터 배포 로그까지, Agent가 다 볼 수 있어야 한다.

진짜 할 일이 많다.

그러나 만약 잘 만들면 놀라운 일이 일어난다. 목표 점수를 통과할 때까지 에이전트는 멈추지 않는다. 계속 피드백 받고 재시도하면서, 창의적인 방법으로 점수를 높여낸다. 채점 도구도 기준도 없을 때 적당히 만들어온 결과물과는 차원이 다르다.

또 한명의 아이돌, 안드레이 카파시도 이렇게 말했다.
"LLMs are exceptionally good at looping until they meet specific goals and this is where most of the 'feel the AGI' magic is to be found."

스스로 피드백을 되먹일 수 있는 채점 시스템.
그게 루프와 마법의 비결이다.

그래서 요즘 쏟아지는 팁 속에서 필요한 것만 흡수해서, 내 채점 시스템을 촘촘하게 만드는 데 집중한다.

구체적으로 뭘 하냐면. 일단 계속 AI와 같이 작업한다. 결과물이 내 기대의 Gap을 찾는다. "이번 세션에서 내 기대와 네 결과물이 달랐던 부분 정리해줘." 그리고 읽으면서 생각한다.

'이 Gap을 다음번엔 자동으로 채점할 수 있을까?'
'어떤 도구로? 어떤 기준으로?'

일부러 시간을 더 써서 채점 시스템을 만들어둔다.

이걸 반복하면 검증 시스템이 촘촘해지고, AI가 더 많은 도구를 쓸 수 있게 된다. 곧 토큰을 10배, 100배 태울 수 있는 구조가 되지 않을까?

Q: 이 Gap을 채점 시스템으로 만드는 과정에서 가장 어려운 부분은 뭐야?

A: • 대부분의 일에서,  제대로 되었는지, 잘 수행되었는지를 평가하는 지표가 '정량적으로 측정가능하고, 빠르게 측정가능하고, 정확하게 우리가 바라는 목적을 반영' 해야한다는 조건을 만족하기가 어렵지.
• 예를 들어 Software의 code quality를 계속해서 높여가는 coding agent라고 해보자. 채점 시스템에서는 뭘 평가해야할까? 굉장히 주관적이야. 결정적이고 단일하며 목적과 일치하는 지표를 찾기가 어렵지.

Q: 이전에 'AI 에이전트가 구현을 담당하게 되면 개발자의 주요 병목 지점은 검증과 리뷰로 이동한다'고 했는데, 그 관점에서 보면 코드 퀄리티 같은 주관적 지표를 어떻게 객관적인 채점 시스템으로 만들 수 있을까?

A: • '코드 퀄리티'도 중요하지만 나는 그보다 '요구사항을 제대로 버그없이 구현했는지'를 채점 시스템으로 만들고 싶어.
• '요구사항을 버그없이 구현했는가'에는 여러가지 채점이 포함될 거야.
    ◦ type check, Linting, Code review by a separate agent, AST-level checks, Tests pass — unit, integration, E2E
    ◦ - Dev server starts without errors — no crash on boot
    ◦ - Console is clean — no uncaught exceptions, no warnings
    ◦ - Visual preview — the UI renders correctly (Claude Code Preview lives here)
    ◦ - Smoke test of the actual user flow — not just "tests pass" but "the feature works end to end"
• 그리고 더 나아가서 궁극의 채점 시스템은, 결국 실제 유저가 다양한 상황에서 이 요구사항의 구현체를 사용할 때 사용에 아무런 문제가 없어야 한다'이겠지. 하지만 그 상황을 정확하게 시뮬레이션해서 채점하기는 어려우니까 우리는 여러가지 proxy 지표들을 쓰는 거야.

Q: 이전에 'AI 에이전트가 구현을 담당하게 되면 개발자의 주요 병목 지점은 검증과 리뷰로 이동한다'고 했던 생각과 연결해보면, 이런 proxy 지표들을 점점 촘촘하게 만드는 과정이 결국 개발자의 역할을 어떻게 바꾸게 될 것 같아?

A: • 개발자는 직접 검증하는 사람이 아니라, 검증의 지표와 도구를 설계하는 사람이 되겠지.

Q: 이런 변화가 개발자로서의 정체성이나 만족감에 어떤 영향을 줄 것 같아?

A: • 그 질문 보다는 나는 이 검증 시스템과 반복을 통한 발전이 영화 &lt;엣지 오브 투모로우&gt;와 비슷하다는 점 말하고 싶어.

Q: 오, 그 비유가 재밌네! 어떤 점에서 엣지 오브 투모로우 같다고 생각해?

A: • 엣지 오브 투모로우의 주인공 빌 케이지는 사실 전투 경험이 전혀 없는 장교야. 하지만 그는 전쟁터로 들어가서 외계인을 마주치는 덕분에, 똑같은 전투를 수십 수백번 반복하게 되지. 
• 그러면서 그는 마치 AI가 강화학습을 해나가듯이 '생존과 적의 사살'이라는 채점 시스템을 가지고 무한히 게임을 반복하면서 전투 기술을 배워나가. 죽어도 다시 시작. 죽어도 다시 시작. 누군가 이렇게 싸우라고 가르쳐준 게 아닌데도 나중에는 전투를 잘하게 되지.
• 그런 것처럼 AI도 계속 반복할 수 있는 환경과 명확한 채점/피드백이 있으면 빌 케이지처럼 될 수 있다는 거야. 

Q: 지금까지 얘기를 한 문장으로 정리하면?

A: AI의 잠재력을 극대화하려면 채점 시스템과 반복 가능한 환경을 설계해서 AI가 유의미한 학습 루프를 돌 수 있도록 만드는 것이 key이다.
