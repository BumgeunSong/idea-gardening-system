Q: 🌱 멀티 Agent를 통한 마켓 시뮬레이션 — 이 주제에 대해 어떤 생각이 있어?

A: • AI가 계속해서 학습을 하기 위해서는 끊임없는 시도와 그에 대한 피드백 데이터가 필요하다.
• deepmind의 genie나  Yann LeCun의 AMI Labs (Advanced Machine Intelligence) 같은 곳은 ‘월드 모델’ 실험 중.
• 세상이 어떻게 작동하는지를 내부적으로 모델링하는 AI. 텍스트 예측이 아니라 현실의 인과관계를 이해하는 시스템.
• LLM은 “다음 토큰이 뭐가 올까?“를 예측합니다. World Model은 “내가 이 행동을 하면 세상이 어떻게 변할까?“를 예측합니다.
• Deepmind가 2025년 8월에 발표한 Genie 3가 그 구체적 결과물.
• Genie 3는 2025년 8월 공개된 최초의 실시간 인터랙티브 범용 world model로, 하드코딩된 물리 엔진 없이 스스로 세상이 어떻게 동작하는지를 학습
• 월드 모델을 소프트웨어 개발에 적용한다면 어떨까?
• 지금 AI 코딩 시스템의 채점 기준이 뭔가?
• 현재 Claude Code, Cursor, Devin 같은 시스템들의 암묵적 reward signal:
• 	∙	빌드가 되는가
• 	∙	테스트가 통과하는가
• 	∙	린트 에러가 없는가
• 	∙	사람이 “좋아요” 눌렀는가 (RLHF)
• 이게 다입니다. “실제로 유용한가”는 측정하지 않아요.
• 결과적으로 AI가 최적화하는 방향이 “기술적으로 올바른 코드”이지 “사람들이 쓰고 싶은 소프트웨어”가 아닙니다.
• 우리가 유저의 집단을 시뮬레이션 할수 있다면?
• 사용자 마켓 시뮬레이션이 있다면, 채점 기준이 이렇게 바뀝니다:
• 지금: 코드가 spec을 만족하는가
• 미래: 이 기능을 실제 유저 1000명에게 보여줬을 때 retention이 올라가는가
• 이건 엄청난 차이입니다. 왜냐면:
• 	∙	지금은 사람(PM, 개발자)이 spec을 작성하고 AI는 그걸 구현합니다. spec 자체가 틀릴 수 있는데 AI는 그걸 모릅니다.
• 	∙	시뮬레이션이 있다면 spec 단계 자체를 검증할 수 있어요. “이 기능을 만들기 전에, 시뮬레이션 유저들이 이걸 원하는지부터 테스트해봐.”

Q: 그런데 왜 "유저 시뮬레이션"이 월드 모델의 자연스러운 다음 단계라고 생각했어?

A: • 월드 모델의 아이디어는 실제 세상과 비슷한 것을 만들어서 AI가 학습하게 하는 것이지.
• 자연스러운 다음 단계가 아니고, 아이디어를 빌려와 본 거야.
• 소프트웨어의 궁극적인 목적 함수는 실제 세상의 유저가 소프트웨어를 사용하면서 만족을 느끼는 것이니까. 
• AI가 어떤 소프트웨어를 만들었을 때, 유저 집단의 만족을 채점할 수 있다면 그쪽으로 AI를 더 학습시킬 수 있을 거야.

Q: 이전에 AI 코딩의 한계에 대해 생각했던 적이 있네. 그런데 "유저 만족"이라는 목적 함수를 측정하려면 어떤 구체적인 시뮬레이션 메커니즘이 필요할 것 같아?

A: • 이 시뮬레이션을 만들려면 결국 “인간이 소프트웨어를 사용하는 방식”을 학습한 모델이 있어야 합니다. 그게 바로 앞서 얘기한 사회적 world model이 있어야 해.
• 그 데이터가 부분적으로 존재합니다:
• 	∙	A/B 테스트 결과 수백만 건 (어떤 UI가 더 전환율이 높았나)
• 	∙	앱스토어 리뷰 (사람들이 실제로 뭘 좋아하고 싫어하는지)
• 	∙	사용자 행동 로그 (어디서 막히고, 어디서 이탈했는지)
• 	∙	Product Hunt 반응 (어떤 기능이 왜 흥분을 일으켰는지)
• 이걸 활용해서 LLM 기반의 가상 유저 모델을 만들 수 있지 않을까?

Q: 어제 AI 코딩의 한계에 대해 생각했던 것과 연결되는데, 이 가상 유저 모델이 실제로 구현된다면 현재 AI 코딩 시스템의 어떤 구체적인 문제를 가장 먼저 해결할 수 있을 것 같아?

A: • AI 코딩의 한계는 스스로 소프트웨어에서 무엇을 개선해야하는지 (what to build) 를 모른다는 거야. 
• 또는 추상적인 spec을 주었을 때 (“댓글 시스템을 만들어봐”) 유저가 만족할만한 댓글 기능을 만들기 위해 구체적으로 문제를 쪼개고 framing하는 게 아니야. 그냥 가장 그럴듯한 다음 빈칸을 채운다는 문제점이 있지.
• 이걸 해결하는 보조적인 시스템이 되지 않을까? 
• 더 나아가면, 소프트웨어 개발의 자율주행 상태“(software factory") 구현에 필요할 것 같아.

Q: 어제 AI 코딩의 한계에 대해 SPEC-TEST 연속체를 논했던 것과 연결되는데, 이 가상 유저 모델이 SPEC 단계에서 가장 큰 변화를 가져올 것 같아. 그런데 "소프트웨어 팩토리"라는 비전을 한 문장으로 표현한다면 뭐라고 할래?

A: Software factory는 마치 로봇만 일하는 dark factory 처럼, 완전 무인으로 돌아가는 소프트웨어 개발 시스템이야. 

StrongDM이라는 회사가 올린 블로그 글에서 따왔어. “We built a *Software Factory*: non-interactive development where specs + scenarios drive agents that write code, run harnesses, and converge without human review.”
